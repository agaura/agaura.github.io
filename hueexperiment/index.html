Hue Experiment

Posted at — Aug 25, 2023

Introduction

  When looking through the various “uniform” color spaces, a common problem occurs: I am 99% certain the hues are not uniform. I feel like I’m being gaslit. In what world is yellow the opposite color of blue? In what world are red, green, and blue equally spaced? Is it not obvious that green and blue are similar, while red is much farther from the other two (particularly from green)?

  Now, it could be the case that this is a remnant from color education when I was a kid. We are taught that the primaries are yellow, red, and blue, and now it is either red, green, and blue or yellow, cyan, and magenta. However… I’m not convinced the original system was wrong. Green and blue still feel so similar to me. How do I prove that I’m right?

  Luckily, I was building a model for a separate post (trying to discuss how lightness differentiation is far superior than hue differentiation and the implications of that), and while building the model I discovered something interesting: when I dither hues in OkLch space it takes a considerable range before I can really tell them apart and percieve the dithering as moving static rather than a solid (or maybe almost solid) color. For instance, at hue = 0°, I only see motion around 70°. That is a huge distance! Not only that… this number varies wildly across the hue spectrum. For instance, at 65°, I only see motion around 100°, a difference half that of when hue = 0°!

  Now, if it were a uniform color space, we’d expect the hue at which the contrast is great enough to percieve motion to be the same for all hues. Instead, it varies wildly, indicating nonuniformity! The best part? I can very easily turn this into an experiment and create a new model based off of my own data.

Intent

  We are trying to create a function which adjust the hues of the OkLch, one of the most successful color spaces for uniformity. Here, we assume:

  • Hues of the same chroma genuinely have the same chroma
  • Hues of the same lightness genuinely have the same lightness

  The latter point is particularly important because lightness is a dimension the eye is very sensitive too, so it could mess with contrast. While it’s hard to prove that OkLab is perfect… when eyeballing it it does seem pretty darn good. Definitely within “I can’t see anything wrong range,” which is not the case for hue itself. In these experiments, we want to build our new hue function by taking data that is solely dependent on hue contrast, as a uniform space should have uniform contrast no matter which spacing of colors on the color wheel you have.

Setup

  For this experiment, the goal is simply to read text on a screen. The trick is, this text barely differs from its background. Every pixel in the background is dithering at a rate of 60hz randomly between two colors: the base hue and the second hue. The text looks identical, except it does not dither over time, so it is absolutely still. The text itself is randomly generated, and the goal is simply to type the text.

  The advantage of this approach is that the contrast threshold $C$ (the difference between the two hues) for when the text is legible is surprisingly high and variable across base hues (often between 25° and 75° of difference between the base and second hue), making it easy to have a standard ruler to relate hues which are not so close to each other on the wheel. This is in contrast to, say, a background with the base color and text with the second color. This second color may just be 5° different from the base before becoming noticeable, maeaning small errors can actually be very significant. This is not to mention that computers can only display so many colors, so it’s better to be working in a higher range where small errors matter less.
